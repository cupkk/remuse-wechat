## 1. 项目概述

### 1.1 产品名称

Re-Museum，数字再生博物馆。

### 1.2 一句话定义

这是一个围绕“旧物上传、AI 识别、故事归档、生成再生成果、进入个人展馆”展开的产品。小程序首期目标是把 Web 端最核心的用户闭环压缩为更适合移动端的前端体验。

### 1.3 本文档的真实用途

这份 PRD 不是给完整 Web 系统立项用的，而是专门为了以下目的编写：

- 明确微信小程序首期前端范围
- 指导 Figma Make 生成小程序界面
- 为后续小程序代码实现提供页面与交互依据
- 帮助后续后端改造时聚焦首期必需能力

### 1.4 小程序首期定位

小程序首期不是把当前 Web 全量搬过去，而是只保留最核心、最适合移动端的体验：

- 登录进入
- 上传旧物
- AI 识别
- 编辑归档草稿
- 保存入馆
- 浏览个人展馆
- 生成与查看贴纸成果

## 2. 背景与问题定义

### 2.1 背景

很多旧物本身并不昂贵，但承载着强烈的生活记忆，例如：

- 奶茶杯盖
- 票根
- 徽章
- 磁带丝带
- 纽扣
- 玻璃瓶

这些旧物通常会被：

- 随手丢弃
- 拍照后遗忘
- 放在角落里没有后续整理

Web 版 Re-Museum 已经证明“旧物 -> 归档 -> 再生成果 -> 展馆”的路径成立，但桌面体验不适合直接照搬到小程序。小程序需要更轻、更直觉、更聚焦主链路的前端设计。

### 2.2 目前要解决的问题

这次小程序前端设计主要解决以下问题：

1. 当前 Web 结构较重，不适合直接迁移到移动端。
2. 小程序首期需要一个清晰的 MVP 范围，避免功能过多导致界面分散。
3. 需要先把前端界面、页面结构、状态设计定下来，再决定后端接口如何适配。
4. 需要一份适合 Figma Make 理解的产品文档，而不是技术实现文档。

## 3. 产品目标

### 3.1 产品目标

- 将 Web 端核心主链路重构为微信小程序可用的移动端体验。
- 让用户在手机端以尽可能低的操作成本完成旧物归档。
- 将归档结果自然衔接到个人展馆和贴纸成果。
- 为后续接入真实后端保留清晰的数据与状态边界。

### 3.2 设计目标

- 界面风格延续 Web 的视觉语言，但不照搬桌面布局。
- 所有页面必须适配微信小程序的触控和单列结构。
- 重点强化空态、加载态、成功态、失败态、权限态。
- 所有关键按钮和卡片必须服务于主链路，而不是仅做装饰。

### 3.3 首期成功标准

若以下目标完成，则认为小程序前端首期设计达标：

- 用户能在 3 到 5 个页面内理解产品是什么
- 用户能顺畅完成上传、识别、编辑、保存的主流程
- 展馆页能清楚承接归档结果
- 贴纸成果有明确的查看和继续操作入口
- 整体视觉统一，有品牌感，不像模板化小程序

## 4. 用户角色

小程序首期只定义 2 类用户角色，不设计邮箱认证流程。

### 4.1 游客登录用户

定义：

- 用户无需绑定正式账号即可快速进入体验
- 适合首次访问、快速试用和体验扫描归档主链路

目标：

- 降低首次进入门槛
- 尽快让用户看到产品价值

设计要求：

- 首页和登录态要明显支持“先试试”
- 游客状态下不要制造复杂账号流程
- 如后续需要升级账号，应保持流程自然

### 4.2 微信登录用户

定义：

- 用户通过微信登录进入正式使用状态
- 适合长期归档、持续使用展馆和贴纸成果

目标：

- 让用户拥有稳定、长期的个人内容空间
- 为后续数据同步和完整功能开放做准备

设计要求：

- 登录方式要明确、可信、低操作成本
- 微信登录应被视作正式主入口
- 登录成功后应直接进入核心产品流程

### 4.3 不在本次前端 PRD 范围内的角色

以下角色或系统虽然在 Web 版中存在，但不属于本次小程序前端设计范围：

- 邮箱注册 / 邮箱验证用户
- 管理员
- 独立运营后台角色

## 5. 目标用户与使用场景

### 5.1 目标用户

首期小程序主要面向以下用户：

- 喜欢保留旧物、纪念物和生活碎片的人
- 对轻量整理和收藏有需求的人
- 喜欢拍照记录并愿意做情绪化表达的人
- 喜欢把日常物件变成贴纸、纪念内容的人

### 5.2 核心使用场景

#### 场景 A：随手记录旧物

- 用户在家里、宿舍、办公桌前看到一个旧物
- 打开小程序
- 直接拍照或选图
- 获得 AI 初步识别结果
- 编辑后保存进个人展馆

#### 场景 B：回看与整理已有收藏

- 用户打开小程序回看自己已经归档的内容
- 进入展馆浏览近期归档结果
- 进一步筛选、查看详情或生成贴纸

#### 场景 C：把旧物变成可分享成果

- 用户选中一件已经归档的旧物
- 生成贴纸成果
- 在结果页查看、保存或继续使用

## 7. 小程序产品价值主张

### 7.1 对用户的核心价值

- 旧物不只是“丢不掉的东西”，而是可以被重新整理和命名的个人记忆。
- AI 不只是生成图，而是帮助用户把旧物变成有结构、有故事的归档内容。
- 归档后的内容不会沉底，而是继续进入展馆和贴纸成果系统。

### 7.2 对首期体验的核心要求

- 简单
- 清晰
- 有情绪氛围
- 主链路强
- 每一步都知道下一步做什么

## 8. 信息架构

### 8.1 小程序一级页面建议

首期建议的小程序一级页面如下：

- 登录页
- 首页
- 扫描页
- 展馆页
- 个人中心页

### 8.2 二级页面建议

- 藏品详情页
- 贴纸结果页 / 贴纸库页

### 8.3 页面关系

建议页面关系如下：

1. 用户进入登录页
2. 登录后进入首页
3. 首页引导进入扫描页
4. 扫描页完成归档后跳转展馆页或藏品详情页
5. 展馆页进入藏品详情页
6. 藏品详情页进入贴纸生成与查看页
7. 个人中心页提供账号状态和基础设置入口

### 8.4 导航建议

建议使用底部 Tab + 页面内跳转的结构。

底部 Tab 推荐为：

- 首页
- 扫描
- 展馆
- 我的

贴纸库可作为：

- 展馆内入口
- 藏品详情内入口
- 或“我的”中的成果入口

## 9. 核心用户旅程

### 9.1 首次进入旅程

1. 用户打开小程序
2. 进入登录页
3. 选择“游客登录”或“微信登录”
4. 登录成功后进入首页
5. 在首页点击进入扫描
6. 上传图片
7. 查看 AI 识别结果
8. 补充或修改故事草稿
9. 保存归档
10. 跳转展馆查看归档结果

### 9.2 回访旅程

1. 用户再次进入小程序
2. 直接进入首页或展馆页
3. 查看最近归档
4. 打开藏品详情
5. 继续生成贴纸或回看成果

### 9.3 贴纸成果旅程

1. 用户进入某件藏品详情
2. 点击生成贴纸
3. 等待生成
4. 查看结果
5. 保存到贴纸库或后续继续使用

## 10. 页面需求明细

### 10.1 登录页

#### 页面目标

让用户快速进入产品，不让账号流程成为第一道门槛。

#### 页面内容

- 产品名称
- 产品一句话说明
- 游客登录按钮
- 微信登录按钮
- 简短的价值说明

#### 页面状态

- 默认态
- 登录中
- 登录失败
- 首次登录成功

#### 设计要求

- 页面不宜过重
- 要明显体现“可先试用”
- 微信登录按钮需突出但不压制游客入口

### 10.2 首页

#### 页面目标

告诉用户这个产品是什么、现在能做什么，以及下一步应该去哪。

#### 页面内容

- 主视觉区
- 产品价值文案
- 进入扫描主按钮
- 最近归档预览
- 展馆入口
- 贴纸成果入口

#### 页面状态

- 首次进入空态
- 有最近归档时的内容态
- 加载态
- 数据获取失败态

#### 设计要求

- 首页更像“品牌入口 + 分发页”
- 必须有明显的扫描入口
- 不能堆过多运营信息

### 10.3 扫描页

#### 页面目标

承载最关键的主链路：上传、识别、编辑、保存。

#### 页面内容

- 拍照入口
- 相册选图入口
- 图片预览区
- AI 识别状态区
- 标题 / 分类 / 材质 / 标签 / 故事草稿编辑区
- 保存归档按钮

#### 页面状态

- 未上传空态
- 图片上传中
- AI 识别中
- 识别完成待编辑
- 保存中
- 保存成功
- 保存失败

#### 设计要求

- 所有状态必须可见
- 用户输入不能在识别或保存时丢失
- 保存成功后要给出明确下一步，例如“查看展馆”或“继续生成贴纸”

### 10.4 展馆页

#### 页面目标

承接归档结果，让用户感受到“我已经拥有一个自己的数字展馆”。

#### 页面内容

- 展馆标题 / 概览
- 馆层或分类切换
- 藏品列表
- 最近新增提示
- 藏品卡片入口

#### 页面状态

- 完全空态
- 仅有少量藏品
- 多分类内容态
- 加载态
- 列表失败态

#### 设计要求

- 卡片浏览要适合移动端滑动
- 展馆不是后台表格，要有氛围感和收藏感
- 筛选切换必须简单直觉

### 10.5 藏品详情页

#### 页面目标

让单件藏品成为“查看、回顾、再生成果”的中心页。

#### 页面内容

- 藏品主图
- 封面图或视觉展示位
- 名称、分类、材质、标签、故事
- 归档时间
- 生成贴纸入口

#### 页面状态

- 数据加载中
- 正常内容态
- 无法加载 / 已删除

#### 设计要求

- 详情页要有沉浸感，不只是字段列表
- 贴纸生成入口应明显但不喧宾夺主

### 10.6 贴纸结果页 / 贴纸库

#### 页面目标

把“归档后的可见成果”明确展示出来，强化成就感和复用感。

#### 页面内容

- 生成结果预览
- 结果说明
- 保存结果入口
- 返回藏品详情或继续浏览入口
- 已有贴纸库列表

#### 页面状态

- 生成中
- 生成成功
- 生成失败
- 无贴纸空态
- 已有贴纸列表态

#### 设计要求

- 强调成果完成感
- 明确区分“正在生成”和“已生成”
- 若生成失败，应给出重新生成入口

### 10.7 个人中心页

#### 页面目标

承载账号状态、基础信息和部分成果入口。

#### 页面内容

- 用户身份状态
- 游客 / 微信登录标识
- 我的归档数量
- 我的贴纸数量
- 成果入口
- 退出登录

#### 页面状态

- 游客态
- 微信登录态
- 数据加载态

#### 设计要求

- 页面应简洁，不承担复杂设置中心角色
- 更像“状态总览页”，不是后台

## 11. 组件与设计系统需求

### 11.1 推荐复用组件

- 底部导航
- 顶部页面头部
- 藏品卡片
- 状态标签
- 主按钮 / 次按钮
- 图片上传容器
- 空态模块
- 加载模块
- 错误提示模块
- Toast / 成功提示

### 11.2 视觉风格要求

视觉上延续当前 Web 产品的基调，但必须适配小程序移动端。

#### 颜色方向

- 深色底
- 荧光黄绿色强调
- 冷青色辅助
- 局部暖色用于成果和情绪点缀

#### 气质关键词

- 博物馆感
- 终端感
- 夜间展厅感
- 数字档案感
- 情绪化但克制

#### 风格限制

- 不要做成模板化 SaaS 后台
- 不要照搬桌面多栏布局
- 不要用太多拟物和复杂特效
- 不要大面积纯白底

### 11.3 排版要求

- 中文为主
- 标题要有品牌感
- 正文要保证移动端可读性
- 卡片信息层级必须清楚

## 12. 关键交互与状态要求

### 12.1 状态设计原则

所有关键异步流程都必须具备完整状态：

- 空态
- 加载态
- 成功态
- 失败态
- 权限态

### 12.2 扫描链路状态

扫描链路至少需要以下状态设计：

- 等待上传
- 图片已选中
- AI 正在识别
- 草稿可编辑
- 正在保存
- 保存成功
- 保存失败

### 12.3 展馆链路状态

- 没有藏品时要引导用户去扫描
- 有少量藏品时要突出精选展示
- 有较多藏品时要支持分类切换

### 12.4 贴纸链路状态

- 未生成
- 正在生成
- 生成成功
- 生成失败
- 已有结果

## 14. 技术与实现边界

### 14.1 本 PRD 聚焦前端，不直接定义完整后端改造

这份文档的重点是小程序前端界面设计，因此：

- 不展开详细数据库设计
- 不展开完整接口文档
- 不展开部署文档
- 不展开后台管理需求

### 14.2 但前端设计需要预留的后端能力

虽然这份 PRD 不做后端细化，但前端设计必须预留这些接口能力：

- 游客登录
- 微信登录
- 上传图片
- AI 识别旧物
- 保存归档
- 获取展馆列表
- 获取藏品详情
- 生成贴纸
- 获取贴纸库

### 14.3 资源访问注意事项

由于原系统的上传资源目前受保护，后续小程序接入时需要特别考虑：

- 图片资源如何展示
- 用户鉴权如何处理
- 上传方式如何适配微信小程序

这些属于后端联调阶段处理事项，但前端界面设计时应考虑“加载中”“资源失效”“重试”等场景。

## 15. 非功能需求

### 15.1 易用性

- 所有核心操作应在单手使用下可完成
- 页面信息层级要清楚
- 主链路按钮必须明显

### 15.2 一致性

- 所有页面使用统一设计语言
- 同类卡片、标签、按钮样式保持一致

### 15.3 可扩展性

- 页面结构应支持后续增加更多成果类型
- 页面布局应支持后续接入真实接口

### 15.4 中文文案质量

- 所有用户可见文案统一使用简体中文
- 避免技术术语直接暴露给普通用户
- 避免出现乱码、英文占位和无意义提示

## 16. 设计协作要求

### 16.1 适合 Figma Make 的工作方式

建议按以下顺序推进：

1. 先确认页面地图
2. 再确认设计系统
3. 然后只做 3 个核心页面
4. 再补页面状态
5. 最后补剩余页面

### 16.2 首批建议优先生成的页面

- 首页
- 扫描页
- 展馆页

### 16.3 设计稿输出要求

- 每个页面必须有明确标题和目标
- 每个关键状态必须单独表达
- 所有核心组件应可复用
- 使用真实感中文文案，不要只写 placeholder

## 17. 明确的非目标

以下内容不是本次小程序前端设计目标：

- 全量复刻 Web 页面
- 实现管理员后台
- 上线 NFC 子站
- 上线灵感广场
- 上线邮箱注册与邮箱认证
- 在首期内实现所有工坊能力

## 18. 小程序首期建议版本范围

### 18.1 MVP 范围

- 游客登录 / 微信登录
- 首页
- 扫描归档
- 展馆浏览
- 藏品详情
- 贴纸生成与结果查看
- 个人中心

### 18.2 第二阶段可考虑

- 贴纸库完善
- 改造指南轻量入口
- 更丰富的展馆筛选

### 18.3 后续阶段再考虑

- 记忆对话
- 共建藏馆
- 拼豆工坊
- 更复杂的成果编辑

## 19. 总结

这份 PRD 的目标不是定义 Re-Museum 的全部系统，而是帮助团队先把微信小程序首期前端界面设计做对。

核心结论只有三点：

1. 小程序首期必须聚焦“登录 -> 扫描 -> 归档 -> 展馆 -> 贴纸成果”这条主链路。
2. 用户角色只区分“游客登录”和“微信登录”，不设计邮箱认证流程。
3. NFC 子站、灵感广场等非核心模块不进入本次小程序前端设计范围。
