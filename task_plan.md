# Re-Museum 项目理解与剩余任务计划

## 目标

深入理解当前 Re-Museum 全栈项目，梳理已经开发的能力、待开发任务、主要不足与改进方向，并形成后续可执行计划。当前最高优先级是保证现场体验链路稳定：

游客进入 -> 拍照上传 -> AI 识别物品特征和故事 -> 推荐一种再生方式 -> 生成成果 -> 展馆 / 我的日历手账浏览。

## 阶段

1. 项目结构与运行方式梳理 - complete
2. 前端主链路与页面状态梳理 - complete
3. 后端 API / 数据 / AI 生成链路梳理 - complete
4. 当前已完成能力与缺口归纳 - complete
5. 形成后续任务优先级与执行顺序 - complete

## 当前约束

- 不修改 `dist/`、`build/` 构建产物。
- 不提交真实 `data/`、`uploads/`、`backups/` 产物。
- 用户可见文案使用简体中文。
- 中文文案改动后必须跑 `fronted npm run check:encoding`。
- 前后端分别构建：`cd fronted && npm run build`、`cd backend && npm run build`。
- 本轮以项目梳理和计划为主，除规划文件外不做功能代码改动。

## 已确认事实

- 仓库没有根 `package.json`，前端在 `fronted/`，后端在 `backend/`。
- 前端依赖 React + Vite + Three.js，后端依赖 Express + SQLite + Google GenAI。
- 当前 `git diff --stat` 为空，最新功能代码很可能已落在工作树基线里；仅 `task_plan.md`、`findings.md`、`progress.md` 是本轮新增规划文件。

## 后续执行顺序

### P0：现场演示主链路

1. 修展馆无藏品时 3D 空卡片仍能旋转的问题 - complete
2. 修展馆详情的生成入口：不能直接跳空结果页，应调用真实生成 loading -> 结果 - complete
3. 强化拍照页体验：允许无故事继续分析；补“第一层识别反馈”状态；保留故事草稿 - complete
4. 补全生成失败后的重试动作，重试应重新调用同一生成类型 - complete
5. 用真实或测试图片跑通：游客 -> 上传 -> 分析 -> 主推荐 -> 生成 -> 展馆 / 我的日历 - partial，本地 `liveAi:false` 已验证真实失败和重试；成功生成需 live AI 环境。

### P1：社区与资产闭环

1. 发布页接后端，不再只本地跳转广场。
2. 广场详情从真实 post 数据进入，去掉硬编码票根详情。
3. 增加点赞、收藏、一键同款风格传递、交换请求基础接口。
4. 展馆详情展示生成成果，并支持从藏品继续生成新成果。

### P2：生产化与增长能力

1. 语音转文字与故事输入体验。
2. AI 配额、调用审计、结构化日志和更细限流。
3. 今日盲盒历史、成就、会员、设置。
4. 部署脚本、线上 smoke、数据备份/恢复流程。
